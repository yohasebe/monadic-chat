# プライバシーフィルター（個人情報マスキング）

## 概要

プライバシーフィルターは、ユーザーメッセージに含まれる個人を特定可能な情報（PII）を AI プロバイダに送信する前にマスクし、AI の応答に対してマスクされた情報を元の値に復元します。検出処理は別の Docker コンテナ（Microsoft Presidio + spaCy）でローカルに実行され、PII が LLM への往復経路で外部に出ることはありません。

プライバシーコンテナは Monadic Chat の標準コンテナセットに含まれているため、初回起動時から利用可能です。各アプリのセッションごとのトグルは初期状態でオフになっており、その会話でマスキングを有効化するには明示的にオンにする必要があります。

## 利用が向く場面

- 実在する氏名・メールアドレス・電話番号を含むメールや文書を作成するとき。
- 個人情報を含むテキストを翻訳するとき。
- 機密性のある識別子を含むスニペットを、手作業で書き換えずに共有したいとき。

## インストール

プライバシーコンテナは Monadic Chat 起動時に自動的にビルドされ起動します。英語は必須であり常にインストールされます。ベースイメージのサイズは約 1 GB です。

英語以外のコンテンツに対するマスキングを有効にするには、追加の spaCy NER モデルを取り込みます:

1. **Settings → Install Options** を開きます。
2. **Privacy Filter — Additional Languages** セクションを探します。
3. ドイツ語 (de)、スペイン語 (es)、フランス語 (fr)、イタリア語 (it)、日本語 (ja)、オランダ語 (nl)、ポルトガル語 (pt)、中国語 (zh) から必要な言語にチェックを入れます。英語は常に有効で外せません。
4. **Save** をクリックします。Privacy セクションのバッジが **rebuild-needed** に変わります。
5. **Settings → Actions** に切り替えて **Build Privacy** をクリックします（メニュー: **Actions → Build Privacy Container** からも実行可能）。Docker が Stopped 状態のときに有効化されます。
6. ビルドが完了するまで待ちます。追加した言語ごとに spaCy モデル（約 50 MB／言語）がコンテナイメージに焼き込まれます。

チェックを外して再ビルドするまで、追加した言語はコンテナに残ります。言語を外して再ビルドすれば次回ビルド時にイメージ容量が削減されます。

実行時のオプトアウト: `~/monadic/config/env` に `PRIVACY_FILTER=false` を設定すると、プライバシーコンテナは起動されなくなります。`true` に戻すまでトグルは無効化され、ツールチップには「Privacy Filter is disabled.」が表示されます。

## セッションごとのトグル

コンテナをインストールすると、プライバシーフィルターをサポートするアプリでは Session Controls パネルに **Privacy Filter** トグルが表示されます。トグルはセッション開始時に必ずオフです。

- 最初のメッセージを送信する前にオンにすると、そのセッションのマスキングが有効になります。
- 最初のメッセージを送ると、トグルはロックされます。**Reset** を押すか別のアプリに切り替えると編集可能な状態に戻ります。
- 最後の選択はアプリごとに記憶されます。あるアプリに戻ると、そのアプリのトグル状態が復元されます。

トグルが無効化されているときは、ホバーすると理由が表示されます:

- *This app does not support Privacy Filter.* — アプリの MDSL に `privacy` ブロックが宣言されていません。
- *Privacy Filter is disabled.* — 環境変数 `PRIVACY_FILTER=false` が設定されており、プライバシーコンテナが起動していません。
- *Privacy Filter is not installed for this language. Install via Settings → Install Options.* — サイドバーで指定されている `conversation_language` が、現在コンテナに焼き込まれている Presidio 対応言語に含まれていません。対象言語を追加して再ビルドするか、conversation_language をインストール済みの言語に切り替えてください。

## 言語の選択

セッショントグルの有効・無効、およびマスキング言語はサイドバーの **conversation_language** に従って決まります:

| サイドバーの conversation_language | マスキング言語 | トグル |
|---|---|---|
| Automatic | 英語 | 有効 |
| English (en) | 英語 | 有効 |
| インストール済みの非英語言語（ja、de、fr など） | その言語（対応する spaCy NER モデル） | 有効 |
| 未インストールの言語（韓国語など） | — | 無効、ツールチップ表示 |

**推奨**: 英語以外のコンテンツをマスクする場合は、サイドバーで conversation_language を明示的に指定してください。「Automatic」は実際のコンテンツに関わらず常に英語としてマスクするため、文化固有のエンティティ（日本語の氏名 + 敬称トリミングなど）を取りこぼす可能性があります。

日本語には専用の追加処理があります: `lang_used` が `ja` の場合、プライバシーコンテナは PERSON のスパン末尾の敬称（`さん`、`様`、`先生`、`君` など）をトリミングし、placeholder には実際の氏名のみが入るようにします。

## インジケーターとレジストリ

フィルターがアクティブな場合、チャットヘッダーにインジケーターが表示されます:

- **Privacy ready**（灰色のアンロックアイコン）: フィルターはオンですが、まだエンティティはマスクされていません。
- **Privacy ON (N)**（緑のロックアイコンとカウント）: 復元用のプレースホルダーが現在 N 件登録されています。
- **Privacy error**（赤）: プライバシーコンテナへ到達できませんでした。Docker が起動していてコンテナが正常か確認してください。

conversation_language が **Automatic** に設定されている場合、最初の信頼できるユーザー入力がセッションの言語をロックすると、インジケーター横に小さな言語バッジ（例: **🌐 ja**）が表示されます。ロックが確定するまでバッジは表示されません。サイドバーで明示的に言語を選んでいる場合は、選択肢自体に言語が出ているのでバッジは省略されます。

インジケーターをクリックすると **Registry Viewer** が開き、各プレースホルダーと元の値、エンティティの種類（PERSON、EMAIL_ADDRESS など）が一覧表示されます。レジストリはセッションメモリ内にのみ保持され、ディスク上の会話ログには書き込まれません。

## アンマスクハイライト

会話カード（ユーザー入力・アシスタント応答の双方）では、Privacy Filter が現在追跡しているすべての値に下線が引かれます。同じプレースホルダーは常に同じ色を使うため、たとえば同じ氏名はすべてのカードで同じ青い下線になり、別の氏名は別の色で表示されます。背景は常にグレーで、色相は下線にだけ反映されるためレイアウトが派手になりません。

下線にマウスを乗せると、その値が経由したプレースホルダー（例: *Tracked as <<PERSON_1>>*) が tooltip として表示されます。ハイライトはマークダウンレンダリング後に適用され、`<code>` / `<pre>` / `<a>` の子要素はスキップされるため、シンタックスハイライトされたコードブロックやリンク文字列は変更されません。

ハイライトは表示用のみで、LLM への送信内容には影響しません。下線が引かれていてもいなくても、LLM 通信時には同じレジストリ経由のマスキングを通過します。

## マスクされる対象

デフォルトでは次のエンティティタイプをマスクします:

`PERSON`、`EMAIL_ADDRESS`、`PHONE_NUMBER`、`CREDIT_CARD`、`IP_ADDRESS`、`IBAN_CODE`、`US_SSN`、`MEDICAL_LICENSE`、`URL`。

`LOCATION`（「Tokyo」など）や `DATE_TIME`（「Friday」など）といった日常的な表現でノイズを生みやすいタイプはデフォルトから除外されています。

アプリ作者は MDSL でマスク対象を調整できます:

```ruby
privacy do
  enabled true
  mask_types :person, :email, :address  # :address を加えると LOCATION が含まれる
end
```

## Text-to-Speech (TTS) のサニタイズ

Privacy Filter が有効なときは、TTS 再生（カードごとの **Play** ボタン、Auto-Speech 経路、システムの **TTS** テスト）も追跡中の PII をストリーミングバッファと同じ短いラベル（例: 「PERSON 1」「EMAIL ADDRESS 1」）に置換してから音声合成 API に渡します。これにより、クラウド TTS API（OpenAI / Gemini / ElevenLabs など）に原文の氏名・住所・電話番号が渡ることはなくなり、長いメールアドレスを 1 文字ずつ読み上げるような UX も避けられます。

Web Speech API（ブラウザ内蔵合成）も同じサニタイザーを通ります。音声生成自体はローカルで行われますが、合成器に渡すテキストはクラウド版と同一のためカード越しに UX が一貫します。

## Knowledge Base への保存と検索

Privacy Filter と Knowledge Base 保存は**アプリ単位で相互排他**です。Privacy 用のアプリ (Chat Plus / Mail Composer / Translate / Second Opinion) では Knowledge Base に保存できません — これらのアプリでは Save ボタンは表示されたままで disabled (グレーアウト) になり、tooltip で理由を表示します。Privacy Filter session toggle が PF 非対応アプリで「表示されるが disabled」になるのと同じパターンです。retrieval 価値のある会話アプリ (Chat / Research Assistant など) では Privacy Filter を有効化できません。Artifact 中心のアプリ (画像 / 動画 / 図 / 文書ジェネレーター) はどちらの機能も持たず、生成された artifact は `~/monadic/data/` に保存され、これらのアプリでは Knowledge Base パネル自体が非表示になります。完全な対応表は [アプリ別 Privacy / KB 対応表](../basic-usage/basic-apps.md#privacy-kb-by-app) を参照してください。

Privacy 用のアプリで扱った会話を残したい場合は **Privacy Export**（暗号化 + 必要に応じた masked モード）を使ってください — 後述の節を参照。

Browse モーダルでは、エントリのタイトルや source に email / 電話番号らしきパターンが含まれているとき、タイトル前に控えめな ⚠️ アイコンが表示されます。フロントエンドの正規表現ヒューリスティックで O(rows) の軽量チェックですが、現在のアプリ割り当てルールが施行される前に保存された PII を含む可能性のあるエントリを一目で識別できるようにする目的があります。

### `library_search` 検索

Privacy Filter が有効なセッションで `library_search` が呼ばれた場合、取得したスニペットも同じ Privacy Pipeline を通過してから LLM に送信されます。Knowledge Base に含まれる PII はリクエスト時に placeholder にマスクされ、LLM の応答側では既存の streaming-handler 経路で復元されます。

これにより、ユーザーが現在のセッションで Privacy Filter を有効にしているにもかかわらず、保存済みの PII が retrieval 経由で LLM に到達する経路を塞いでいます。マスキングは best-effort で、pipeline が例外を投げた場合は raw snippet にフォールバックして検索自体は失敗させません。

### Global と App-only スコープ

会話を **Global** スコープで保存すると（デフォルトの **App-only** ではなく）、この Monadic Chat インストール内のすべてのアプリから `library_search` 経由で取得可能になります。保存ダイアログでは Global を選んだ瞬間にインライン警告が表示され、まだ選んでいない将来のアプリからも取得できる状態になることをリマインドします。明示的に共有したい会話以外は App-only を選んでください。

## 暗号化エクスポート

プライバシーフィルターが現在のセッションで少なくとも 1 件のエンティティをマスクしている場合、**Export** ボタンは 2 つの直交する軸を持つ統合エクスポートダイアログを開きます:

- **Encryption**: AES-256-GCM（Argon2id 鍵導出）でファイルを暗号化します。パスフレーズはユーザーが指定します。
- **Content**: 元の値を復元した内容で、またはプレースホルダーを残した状態で出力します。

セッション内で Privacy がアクティブな場合、ダイアログのデフォルトは **Masked** になっています。意図せずに平文 PII をディスクに書き込まないための安全側デフォルトです。**Restored** を明示的に選んだ上に暗号化を OFF にした組み合わせは、赤い強警告バナーでファイルに何が書き出されるかを説明します。平文 PII を含むファイルにはファイル名末尾に `-PRIVATE` サフィックスが付くため、ダウンロードフォルダの中で識別しやすくなっています。

代表的な組み合わせ:

1. **Encrypted + Restored** — 自分以外には開けない個人アーカイブ。
2. **Encrypted + Masked** — 復号後も PII を露出させない解析ログの共有用。
3. **Plain + Masked** — 実名を含まない簡易な共有サンプル。
4. **Plain + Restored** — `-PRIVATE` 付きの最も警告の強いモード。信頼できるローカル限定のワークフロー用。

パスフレーズの要件: 8 文字以上で確認フィールドが一致すること。強度メーターは情報提供のみで、送信を妨げません。

暗号化エクスポートを再度開くには **Import** ボタンを使い、プロンプトに従ってパスフレーズを入力します。レジストリも会話と一緒に復元されるため、続くメッセージで一貫したマスキングが行えます。

## ドキュメント DB のエクスポート / インポート（Actions メニュー）

**Actions → Export Document DB** と **Actions → Import Document DB** メニューは qdrant ボリューム全体（保存されたすべての会話・PDF・Knowledge Base エントリ）を対象とします。両メニューとも、実行前に確認ダイアログを表示するようになりました。

エクスポートでは次の 2 つから選べます:

- **Encrypt and Export**（デフォルト）: パスフレーズを尋ね、tarball を AES-256-GCM（PBKDF2-SHA256, 600 000 回反復）でエクスポートごとのランダムソルトを使って暗号化し、`monadic-qdrant.tar.gz.enc` を共有フォルダに書き出します。暗号化に成功すると平文の tarball は削除されます。
- **Export Plain**: 暗号化せずに tarball を直接書き出します。ダイアログでは、保存済み会話 / PDF が Privacy Filter の利用状態によらず平文で含まれることを警告します。

インポートはファイル拡張子を自動判別します: `.tar.gz` は従来通り展開され、`.tar.gz.enc` ではパスフレーズを尋ねて一時平文ファイルに復号した後 qdrant ボリュームのインポートを実行し、最後に一時ファイルを削除します。パスフレーズが間違っていたり改ざんされている場合は復号に失敗し（tarball は中途半端に展開されません）、qdrant ボリュームには手が触れられません。

暗号化フォーマット: `[magic 'MQDB'] [version 0x01] [salt 16] [iv 12] [ciphertext] [authTag 16]`。ストリーミング処理可能な構造のため、複数 GB のボリュームでもメモリに乗せきる必要はありません。

## 対応アプリ

次のアプリの MDSL 定義でプライバシーフィルターが有効化されています:

- Mail Composer
- Chat Plus
- Translate
- Second Opinion
- Chat

その他のアプリは `privacy do` を宣言していないため、トグルは無効のままです。コードやメディア、音声生成を主目的とするアプリは除外されています。生成されたコード内にプレースホルダーが残るなど、本来の出力にマスキングが干渉してしまうためです。

## 制限事項

- カスタム認識器はプライバシーコンテナのソース（`docker/services/privacy/recognizers/`）に存在します。新しい認識器を追加するにはコンテナの再ビルドが必要です。
- マスキングはメッセージごとにわずかなレイテンシを追加します（テキスト長によりますが通常 50-200 ms 程度）。各ターンで context 全体（user メッセージと過去の assistant メッセージの両方）を再マスクし、registry の整合性を保ちます。
- マスキング対応言語は Presidio に同梱される 9 種類の spaCy NER モデル（英語、ドイツ語、スペイン語、フランス語、イタリア語、日本語、オランダ語、ポルトガル語、中国語）に限られます。これ以外の言語（韓国語、アラビア語など）はプライバシーフィルターの対象外で、サイドバーの conversation_language がそれらに設定されているとセッショントグルは無効化されます。

## 設定リファレンス

`~/monadic/config/env` に保存される設定:

| 変数 | 値 | デフォルト | 説明 |
|---|---|---|---|
| `PRIVACY_FILTER` | `true` / `false` | `true` | 実行時ゲート。`false` に設定するとプライバシーコンテナは起動しません。`true` に戻すまでトグルは無効化されます。 |
| `PRIVACY_LANGS` | カンマ区切りの言語コード | `en` | プライバシーコンテナにビルド時に焼き込まれる spaCy NER モデル。英語は必須で先頭に自動付与されます。推奨される編集経路は Settings → Install Options → "Privacy Filter — Additional Languages"。高度なユーザは env ファイルを直接編集することも可能です。対応コード: `en`、`de`、`es`、`fr`、`it`、`ja`、`nl`、`pt`、`zh`。 |
| `PRIVACY_DEV_PORT` | ポート番号 | `8001` | 開発モード時のみ、プライバシーコンテナの HTTP ポートをホストに公開する用途で使われます。 |

`PRIVACY_LANGS` を変更した後は、コンテナを再ビルド（Settings → Actions → Build Privacy）し、Monadic Chat を再起動して新しい言語モデルを反映させてください。`PRIVACY_FILTER` を変更した場合は、Monadic Chat の再起動のみで反映されます（再ビルドは不要）。
