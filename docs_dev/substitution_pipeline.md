# Substitution Pipeline

This is an internal-developer document describing the architecture and design
decisions behind the **Substitution Pipeline** — the layer that mediates the
symbolic "contact points" shared between the user and the assistant. It unifies
two features that both rewrite text at the user↔AI boundary:

- **Privacy Filter** — masks PII before the LLM sees it (`<<PERSON_1>>`) and
  restores it in the response.
- **Vocabulary** — lets the user and assistant share `${TOKEN}` variables (e.g.
  `${SHARED}` for the synced data folder) that resolve to real paths/state the
  model should not need to know verbatim.

The user-facing description of the Privacy Filter lives in
`docs/advanced-topics/privacy-filter.md`; Vocabulary has no public docs yet
(no app exposes it — see "Status" below).

## Why one abstraction

Both features are instances of the same idea: a *symbol* functions as a shared
contact point so the two sides of a conversation can cooperate without their
internal "understanding" having to match. PII masking and shared-path variables
are different policies over the same mechanism — rewrite on the way in, resolve
on the way out. Putting them on one provider-based pipeline means a third such
feature is "just another provider", and the lifecycle/ordering rules are written
once.

## Terminology

- **Pipeline** (`lib/monadic/substitution/pipeline.rb`,
  `Monadic::Substitution::Pipeline`) — orchestrator holding an ordered list of
  providers; runs them at each lifecycle stage and exposes a token-resolution
  chain.
- **Provider** (`lib/monadic/substitution/provider.rb`) — abstract base. A
  provider overrides lifecycle hooks and/or declares which `${TOKEN}` names it
  owns.
- **Context** (`lib/monadic/substitution/context.rb`) — per-message read surface
  wrapping the session hash + active app.
- **PrivacyFilter provider**
  (`lib/monadic/substitution/providers/privacy_filter.rb`).
- **Vocabulary provider**
  (`lib/monadic/substitution/providers/vocabulary.rb`) + the built-in token
  registry (`lib/monadic/substitution/vocabulary.rb`).

## Two token namespaces (disjoint, both inside `${...}` conceptually)

| Feature | Wire form | Generated by | Owns-token rule |
|---|---|---|---|
| Privacy | `<<PERSON_1>>`, `<<EMAIL_ADDRESS_1>>` … | the Python Presidio container (`docker/services/privacy/server.py`) | `<TYPE>_<N>` |
| Vocabulary | `${SHARED}` … | declared in `Vocabulary::BUILTINS` | single-word UPPER_CASE |

The two never collide. Note the Privacy wire form is `<<TYPE_N>>`, **not**
`${TYPE_N}` — detection and placeholder numbering happen in Python; the Ruby
side only consumes and round-trips that form. (An early design note claiming
Privacy used `${...}` was inaccurate; `${...}` is the Vocabulary syntax.)

## Lifecycle hooks (Provider)

| Stage | Hook | Privacy | Vocabulary |
|---|---|---|---|
| user input | `on_input(text, ctx)` | PII → `<<TYPE_N>>` | no-op (the LLM must SEE `${SHARED}`) |
| system prompt | `system_prompt_addendum(ctx)` | — | "## Shared variables" section |
| tool call | `on_tool_invoke(name, args, ctx)` | no-op | expand `${SHARED}/x` → real path (deep) |
| output render | `on_output_render(text, ctx)` | restore `<<TYPE_N>>` → PII | (retained for contract; live decoration moved to frontend — see below) |

Plus `owns_token?` / `resolve` for the resolution chain, and `failure_mode`
(`:closed` for Privacy = must propagate to prevent leaks; `:open` for Vocabulary
= a miss must never break a turn).

## Frozen design decisions

| # | Decision |
|---|---|
| A | Unified `${TOKEN}` syntax for *Vocabulary* (Privacy keeps its legacy `<<TYPE_N>>` wire form). |
| B | Escape via backtick code wrap: `` `${SHARED}` `` is left literal (not expanded). |
| C | App-local scope: the pipeline is rebuilt on app change. |
| D | A resolver lambda takes the **session hash**. |
| E | Display resolver defaults to the tool resolver. |
| F | Failure policy is per-provider (`failure_mode`). |

## Privacy: refactored, not rewired

Privacy was recast as a `Substitution::Provider` (Phase 2.2) **behavior-
preservingly**. Its live hot path is unchanged: vendor adapters still call
`apply_privacy_to_messages` / `before_send_to_llm` (mask) and
`after_receive_from_llm` (restore) in `streaming_handler.rb`. The class is the
provider, aliased back as `Monadic::Utils::Privacy::Pipeline` so the ~21
duck-typed call sites keep working.

Critical invariant: the registry stays at `session[:monadic_state][:privacy]`
(via the existing `Registry`), **not** the generic `Provider#state`
(`session[:substitution_state]`) slot — `Provider#state` is overridden to raise.
This preserves `strip_for_persist`, the language detector's `[:detection]`
sub-state, and the privacy export/import paths.

Safety net: `spec/golden/privacy/` records the Ruby transformation behavior as
format-neutral golden fixtures replayed through a stubbed (recorded) Presidio
response — deterministic, container-free. See the header of
`spec/golden/privacy/support.rb`.

### Privacy failure policy (audited 2026-06-12)

Direction decides the failure mode:

- **Toward the provider (masking) — fail-closed.** If masking cannot run,
  the unmasked text must not leave the machine.
  - `PrivacyFilter#before_send_to_llm` raises `BackendError` on backend
    failure (`on_failure: :block` is the default; `:pass` is an explicit
    per-app opt-out).
  - `apply_privacy_to_messages` (base_vendor_helper) has no rescue — the
    error propagates and the request fails. Intentional; do not add a
    rescue that returns the original messages.
  - `library_search.apply_privacy` replaces the search results with
    `WITHHELD_MESSAGE` and reports via `DegradationNotifier` when masking
    raises. (Until 2026-06-12 it returned the raw text — i.e. it leaked
    unmasked KB snippets exactly when the privacy backend was broken.)
- **Toward the user (restoring) — fail-open is acceptable.** Restoration
  happens locally (find-and-replace from the registry); if it fails the user
  sees `<<TYPE_N>>` placeholders. Ugly, but no PII leaves the machine, and
  failing the whole turn would destroy the response. `after_receive_from_llm`
  and `deanonymize` therefore fall back to the masked text.

Rule of thumb for new call sites: a rescue on the masking path must never
return the input text; it must withhold (or raise) and report through
`Monadic::Utils::DegradationNotifier`.

## Vocabulary: parser → provider → live wiring → UI

1. **Parser** (Phase 3) — `vocabulary do; use :shared; end` in MDSL.
   `MonadicDSL::VocabularyConfiguration#use` validates each name against
   `Monadic::Substitution::Vocabulary::BUILTINS` (typo fails at load) and
   produces plain data `{ tokens: [:shared] }` (must stay `.inspect`-able for
   the app-class generator in `dsl.rb`). Resolvers live in Ruby, never in MDSL
   (MDSL is declarative).
2. **Provider** (Phase 4) — `Providers::Vocabulary.new(tokens:)`. `on_tool_invoke`
   **expands** owned `${TOKEN}` (and the `${TOKEN}/sub/path` prefix) deeply across
   Hash/Array/String args; backtick-wrapped tokens are left literal (decision B).
   `BUILTINS[:shared].resolve` is `->(session){ Environment.shared_volume }`.
3. **Live wiring** (Phase 5) — `base_vendor_helper.rb` adds
   `substitution_pipeline_for` (opt-in, caches a **Vocabulary-only** pipeline at
   `session[:_substitution_pipeline]`), `expand_tool_args_for_vocabulary`
   (inserted in all 8 vendors' `process_functions`, before `:session` injection so
   the session object is never walked), and the system-prompt addendum (a single
   `:vocabulary_variables` rule in `SystemPromptInjector`'s `SYSTEM_INJECTION_RULES`,
   not 8 vendor edits — `build_injections` already resolves `APPS[app_name].settings`).
   Cleared on app-change/reset in `misc_handlers.rb`.
4. **Token UI** (Phase 6) — decoration moved from backend to frontend. The
   backend ships a `vocabulary_map` (`{ "SHARED" => "/resolved/path" }`) with the
   message (mirroring `privacy_known_entities`); `ws-vocabulary-handler.js` walks
   the rendered DOM (including inside inline `<code>`, where LLM-backticked paths
   live) and wraps `${TOKEN}` in a clickable `.vocab-token` span. Click opens the
   path in the OS file explorer (`shell.openPath`/`showItemInFolder`, cross-
   platform) via the existing `webview-preload.js` → `ipcMain('reveal-path')`
   bridge; browser mode falls back to clipboard copy.

### Why Vocabulary is on its own pipeline (not shared with Privacy)

Privacy is `:closed`, Vocabulary `:open`; they have independent lifecycles and
disjoint namespaces. Co-locating them on one `process_*` fold would entangle the
two failure contracts. Privacy stays on its proven legacy path; the
`:_substitution_pipeline` holds only Vocabulary.

### Dual-mode reveal translation

In production the Ruby backend runs inside the container, so `${SHARED}` resolves
to the container path `/monadic/data` — correct for tool expansion (tools run in
the container). But the OS file explorer runs on the host, where the same data is
at `~/monadic/data`. The fix is at the **exit**: `ipcMain('reveal-path')` in
`app/main.js` maps the `/monadic/data` prefix to `path.join(os.homedir(),
'monadic','data')`. The resolver and the shipped map stay container-correct;
dev-mode paths (already host paths) pass through.

## Status

Live. `${SHARED}` is **on by default for every app** — the policy lives in
`Monadic::Substitution::Vocabulary.tokens_for(app_settings)`, the single source
of truth consulted by both the pipeline builder and the system-prompt injector.
An app opts out with `vocabulary false` in its MDSL. There is no per-app opt-in
to add; `vocabulary do … end` is reserved for future custom tokens.

`${SHARED}` is also **actionable** everywhere: `MonadicDSL.inject_file_operations!`
(dsl.rb, mirrors `inject_library_search!`) gives every app the shared-folder
`read/write/list` tools by default (no exclusions). Consequence: a tool-capable
orchestration model is the baseline — non-tool-capable models simply never
receive the tools (vendor helpers gate on `tool_capability`), consistent with
the existing "tool calling required" stance. End-user docs live in
`docs/docker-integration/shared-folder.md` (+ `docs/ja/...`), "The `${SHARED}`
Variable" section.
